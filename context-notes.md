## 2026-07 — 초기 기획 논의

### 결정 1: 클럽(장비) 추적은 1차 범위에서 제외

- 검토한 대안: (A) 사람 골격만 우선 구축, 클럽은 2차 (B) 처음부터 클럽 포함
- 선택: A
- 근거: 클럽 궤적 추적은 사람 포즈 추정과 완전히 다른 CV 문제(객체 탐지/추적)라 범위가 커짐. 야구 파이프라인 재사용률을 높여 빠르게 출시하는 걸 우선함.
- 참고: 클럽까지 다루는 오픈소스 사례로 `mamoonik/golf-swing`(MediaPipe + 클럽 포함 38 키포인트)을 2차 확장 시 참고.

### 결정 2: 기준값(정상범위) 부재 문제 — 신호등 없이 수치/그래프만 제공

- 배경: 야구는 Driveline OBP처럼 레벨별 통계 데이터가 있어 신호등(정상/주의/위험) 판정이 가능했음. 골프는 조사 결과 이런 공개 데이터셋을 찾지 못함(GolfDB, SportsPose 등은 포즈 추정 정확도 검증용이지 코칭 기준 통계가 아님).
- 검토한 대안: (A) 신호등 없이 수치만 (B) 코치 수동 입력 기준값 (C) 자체 데이터 수집
- 선택: A
- 근거: 근거 없는 신호등을 만드는 것보다 "기준 데이터 없음"을 정직하게 노출하는 게 낫다고 판단. B, C는 리소스/시간이 더 필요해 이번 범위에서 제외.
- 재검토 조건: 코치 대상 기능으로 확장할 계획이 생기면 B안(수동 기준값 입력)을 다시 검토.

### 결정 3: 종목 선택 UX를 "업로드 전 필수"에서 "업로드 후 선택적 필터"로 전환 (야구 포함 전체 적용)

- 계기: 실제 화면 스크린샷 검토 중, 현재 플로우가 "종목 선택 → 업로드" 순서인데, 포즈 추출 자체는 종목 무관 처리라는 점을 지적받음.
- 근거:
  - 포즈 추출(무거운 연산)을 종목 결정과 분리하면, 종목을 잘못 고르거나 나중에 바꾸고 싶을 때 재업로드가 필요 없음.
  - 기존에 있던 "General - Basic static posture" 카테고리가 사실상 "종목 미지정" 상태와 같다는 점에서, 이 구조가 자연스럽게 맞아떨어짐.
  - 앞서 합의한 아키텍처(범용 5개 섹션 + 종목별 추가 섹션)와도 일치함.
- 트레이드오프로 논의된 것: 촬영 가이드(카메라 각도 등)를 종목 선택 이전에 줄 수 없게 됨.
  - 검토한 대안: (A) 범용 가이드만, 종목별 생략 (B) 업로드는 먼저, 종목 선택 후 재촬영 권장 안내 추가 (C) 촬영 방식은 종목 무관하다고 보고 신경 안 씀
  - 선택: A — 종목별 촬영 가이드 문구 작업 자체를 하지 않기로 함
- 종목 미선택(General) 상태 UI: 범용 섹션만 보여주고 끝나는 게 아니라, 언제든 종목을 추가 선택할 수 있는 진입점을 상시 노출하기로 함 (완전히 닫힌 종료 화면이 아님).
- 데이터 모델 영향: `AnalysisRecord`의 `category`를 nullable 선택적 태그로 전환(별도 `sport` 필드는 만들지 않음 — `category`가 이미 스포츠 값을 담고 있어 중복 필드를 만들면 출처가 두 개로 꼬임). 업로드 시점이 아니라 분석 결과 화면에서 붙이는 선택적 태그로 취급.

### 결정 4: 골프 이벤트는 GolfDB 8단계 표준을 그대로 채택

- 근거: 컴퓨터 비전 연구 분야에서 이미 표준화된 정의가 있으므로 야구처럼 처음부터 이벤트를 정의할 필요가 없음.
- 8단계: Address → Toe-up → Mid-backswing → Top → Mid-downswing → Impact → Mid-follow-through → Finish

### 결정 5: 이벤트 자동 감지는 야구의 실패 교훈을 그대로 적용

- 야구에서 얻은 핵심 교훈: 탐색 범위를 퍼센트로 "추측"하는 방식은 근본적으로 불안정하다. 탐색 범위 자체를 사용자가 지정한 구간으로 좁히고 그 안에서만 알고리즘을 돌리는 방식이 훨씬 안정적이었다.
- 골프 적용: 사용자가 Address/Finish 두 프레임만 수동 지정 → 그 구간 안에서만 8단계 후보 자동 탐색 → 후보 프레임을 ±3프레임(총 7개) 썸네일로 제시 → 클릭 한 번으로 확정.
- 감지 알고리즘 자체는 종목마다 다름(야구=골반 x좌표 최대 편차, 골프=손목 궤적 기반 — GolfPosePro 참고). 이 때문에 감지 로직은 종목별 독립 모듈로 분리하고, 하나의 함수에 종목 분기(if sport === 'golf')를 넣는 방식은 피하기로 함.
- 원인 불명확한 오류 대응 원칙도 그대로 유지: 추측성 원인을 실제 확인 없이 방지 코드로 넣지 않는다.

## Phase 0 구현 (2026-07-24) — 종목 선택 UX 리팩터링

- 업로드 화면(`Analyze.jsx`)에서 종목 선택 단계를 제거. 업로드 → 포즈 추출 → 범용 결과 화면까지 종목 없이 동작.
- `AnalysisRecord.category`를 `required`에서 제거해 nullable 선택적 태그로 변경. 업로드 시 `category`는 null로 저장.
- 결과 화면(`FrameAnalysis.jsx`)에 상시 노출되는 "종목 선택" 칩(일반/미지정 · ⚾ 야구 · ⛳ 골프) 추가. 선택 시 해당 레코드의 `category`를 즉시 업데이트해 히스토리에 반영.
- 야구 OBP(투구/스윙 토글 + ObpComparison)는 종목=야구일 때만 노출되도록 게이트(기존 always-on에서 종목 선택 기준으로 변경). 야구 회귀: 야구 선택 시 기존과 동일하게 동작.
- 골프는 칩 선택은 가능하나 이벤트 자동 감지/결과 갤러리는 아직 미구현("곧 추가" 안내만). Phase 1~3에서 구현 예정.
- `MyPage.jsx` 히스토리는 `category`가 null일 때 "일반 자세"로 표시, 레코드 id를 결과 화면으로 전달해 종목 재태깅 지원.
- 정적 이미지 분석(`AnalysisReport`) 흐름도 업로드 시 종목 미지정으로 동작. 정적 이미지의 종목 태그 선택 UI는 이번 페이즈에서 미추가(비디오 흐름이 골프/야구의 중심).
